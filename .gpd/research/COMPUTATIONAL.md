# Computational Approaches: ViT-Based LIGO Glitch Classification

**Surveyed:** 2026-03-16
**Domain:** Gravitational-wave detector characterization / deep learning
**Confidence:** HIGH

## Recommended Stack

Use PyTorch with the `timm` library for Vision Transformer training on Q-transform spectrograms generated by GWpy from LIGO O3 strain data. The Gravity Spy labeled dataset (Zenodo) provides ground-truth glitch classifications across ~22 morphological classes for O1-O3. The full pipeline is: GWOSC bulk strain download -> GWpy Q-transform spectrogram generation -> PNG/numpy array dataset -> ViT fine-tuning via timm/HuggingFace Transformers -> evaluation with scikit-learn metrics.

This stack is mature, well-documented, and fits entirely within a single RTX 5090 (32 GB VRAM) with mixed-precision training. No HPC or multi-GPU infrastructure is needed. The main computational bottleneck is spectrogram generation (embarrassingly parallel, CPU-bound), not model training.

## Numerical Algorithms

| Algorithm | Problem | Convergence | Cost per Step | Memory | Key Reference |
|-----------|---------|-------------|---------------|--------|---------------|
| Q-transform (constant-Q) | Time-frequency decomposition of strain data | Exact (analytic transform) | O(N log N) per segment | O(N_freq x N_time) | Chatterji et al. (2004) |
| AdamW optimizer | ViT weight optimization | Empirical; ~50-100 epochs for fine-tuning | O(P) where P = num params | O(3P) for optimizer states | Loshchilov & Hutter (2019) |
| Cosine annealing LR | Learning rate schedule | N/A (schedule, not optimizer) | Negligible | Negligible | Loshchilov & Hutter (2017) |
| Stratified K-fold CV | Robust evaluation under class imbalance | K iterations | K x training cost | K x model size | Standard |

### Convergence Properties

**Q-transform:**
- **Convergence criterion:** Not applicable -- deterministic transform. Output quality depends on Q-range and frequency resolution parameters.
- **Expected rate:** Single-pass computation.
- **Known failure modes:** Poor frequency resolution if Q-range is too narrow; aliasing if sample rate is insufficient for target frequency band.

**AdamW for ViT fine-tuning:**
- **Convergence criterion:** Validation macro-F1 score plateaus for 10+ epochs (early stopping patience = 10). Use macro-F1 not accuracy because of severe class imbalance in Gravity Spy data.
- **Expected rate:** Validation loss should decrease monotonically for first ~20 epochs with pretrained initialization. Fine-tuning from ImageNet-pretrained ViT-B/16 typically converges in 30-80 epochs on datasets of this size (~10^5 samples).
- **Known failure modes:** (1) Learning rate too high destroys pretrained features -- use lr ~1e-4 to 5e-5 for fine-tuning, with lower lr for transformer blocks and higher for classification head. (2) Batch size too small causes noisy gradients -- use gradient accumulation to achieve effective batch size >= 64. (3) Rare class collapse -- model predicts majority class for all rare glitches.

## Data Flow

```
GWOSC bulk download (HDF5 strain files, 4096 Hz, per detector)
  -> GWpy TimeSeries.read() per glitch trigger time
  -> GWpy TimeSeries.q_transform() at 4 durations (0.5s, 1.0s, 2.0s, 4.0s)
  -> Normalize spectrogram (see Pitfalls: normalization)
  -> Save as PNG (224x224) or numpy array (.npy)
  -> Gravity Spy CSV labels merged by GPS time
  -> Train/val/test stratified split (70/15/15)
  -> PyTorch Dataset + DataLoader with augmentation
  -> ViT-B/16 (timm, pretrained on ImageNet-21k) fine-tuning
  -> Evaluation: per-class F1, confusion matrix, rare-class recall
```

### Data Access: GWOSC Strain Data

**Method:** Use the `gwosc` Python package to discover file URLs, then bulk download with `wget` or `requests`.

```python
# Discover O3a strain files for H1 at 4096 Hz
from gwosc.api import fetch_run_json
run_data = fetch_run_json('O3a_4KHZ_R1', 'H1')
# Returns list of file URLs; download with standard HTTP tools
```

**Alternative:** `gwpy.timeseries.TimeSeries.fetch_open_data('H1', t0, t1)` for on-demand access to specific segments. Use this for prototyping; use bulk download for production.

**O3 strain data size estimate:**
- O3a: Apr 2019 - Oct 2019 (~6 months) + O3b: Nov 2019 - Mar 2020 (~5 months)
- ~11 months of data per detector, 2 LIGO detectors (H1, L1)
- 4096-second HDF5 files at 4 kHz: ~120 MB each
- ~700 files per detector per sub-run
- **Total download: ~300-400 GB for H1+L1 at 4 kHz**
- At 16 kHz: ~4x larger. Use 4 kHz unless frequency content above 2 kHz is needed (it is not for most glitch classes).

**Recommendation:** Download only the 4 kHz data. Most glitch morphologies are well-captured below 2 kHz. This halves storage and speeds up Q-transforms.

### Data Access: Gravity Spy Labels

**Primary source:** Zenodo record 5649212 -- Machine Learning classifications for O1, O2, O3a, O3b. Contains CSV files with columns: `gravityspy_id`, `event_time` (GPS), `ifo`, `label`, `confidence`, and spectrogram image URLs.

**Dataset statistics (O1-O3 combined):**
- ~1.2 million total triggers classified by ML pipeline
- ~10,000 high-confidence labeled training samples (volunteer-vetted)
- 22-24 morphological classes (varies by observing run)
- Severe class imbalance: Blip and Scattered_Light dominate; rare classes (Paired_Doves, Helix) have <100 samples

**Spectrogram images:** Gravity Spy provides pre-made spectrogram images at 140x170 pixels across 4 time durations per glitch. You can either:
1. Use pre-made Gravity Spy images (fast, but locked to their Q-transform parameters)
2. Regenerate from strain data with GWpy (slower, but full control over resolution, frequency range, normalization)

**Recommendation:** Start with pre-made Gravity Spy images resized to 224x224 for rapid prototyping. Then regenerate custom spectrograms from strain data for the final model, as this gives control over normalization and resolution.

### Spectrogram Generation Pipeline

**Input:** Strain data segment centered on glitch GPS time
**Output:** 224x224 pixel spectrogram image (or numpy array)

```python
from gwpy.timeseries import TimeSeries

# Load strain segment around glitch
strain = TimeSeries.fetch_open_data('H1', gps_time - 2.0, gps_time + 2.0)

# Whiten (critical -- removes detector noise curve shape)
white = strain.whiten(fftlength=4, overlap=2)

# Q-transform
qspecgram = white.q_transform(
    frange=(10, 2048),   # Hz -- covers all glitch morphologies
    qrange=(4, 64),      # Q factor range
    outseg=(gps_time - 0.5, gps_time + 0.5),  # crop to duration
    tres=0.002,          # time resolution in seconds
    fres=0.5,            # frequency resolution in Hz
    whiten=False         # already whitened
)

# Normalize to [0, 1] and save as image
import numpy as np
data = qspecgram.value
data = np.clip(data, 0, 25.5)  # clip SNR to reasonable range
data = data / 25.5  # normalize
```

**Four durations per glitch:** Generate spectrograms at 0.5s, 1.0s, 2.0s, 4.0s windows (matching Gravity Spy convention). For single-view ViT, use the 1.0s or 2.0s view. For multi-view, concatenate or use separate channels.

**Parallelization:** Spectrogram generation is embarrassingly parallel. Use Python `multiprocessing` or `joblib` with 12 cores. Expect ~0.5-1.0 seconds per spectrogram per core. For 100,000 glitches x 4 durations = 400,000 spectrograms: ~50-100 CPU-hours, or ~5-10 hours on 12 cores.

### GPU Memory Budget (RTX 5090, 32 GB VRAM)

**ViT-B/16 (86M parameters) with mixed precision (AMP):**
- Model weights (FP16): ~170 MB
- Optimizer states (FP32): ~680 MB
- Gradients (FP16): ~170 MB
- Activations at batch size 64, 224x224: ~4-6 GB
- **Total at batch size 64: ~6-8 GB VRAM**
- **Headroom for batch size 128: ~12-16 GB VRAM**
- **Maximum feasible batch size: ~256 (uses ~24-28 GB)**

This GPU has ample capacity. Use batch size 64-128 with AMP (torch.cuda.amp). No gradient checkpointing or gradient accumulation needed, though gradient accumulation is useful for simulating larger effective batch sizes if needed for training stability.

## ViT Architecture Decisions

**Use ViT-B/16 (Base, patch size 16) pretrained on ImageNet-21k.** Rationale:
- 86M parameters -- large enough for the complexity of glitch morphologies, small enough for single-GPU training
- Patch size 16 on 224x224 images = 196 patches = 196 tokens -- manageable sequence length
- ImageNet-21k pretraining provides strong low-level feature representations (edges, textures) that transfer well to spectrograms
- ViT-L/16 (304M params) would work but offers diminishing returns on a dataset of ~10^5 samples and slows iteration

**Input strategy (single-view vs multi-view):**
- **Start with single-view:** Use the 1.0s duration spectrogram. This is the simplest baseline.
- **Upgrade to multi-view:** Concatenate 4 duration views as 4 channels (modify patch embedding input channels from 3 to 4), or use a late-fusion architecture. The Gravity Spy team's 2024 paper (arXiv:2401.12913) showed multi-view attention fusion improves classification, especially for morphologically ambiguous classes.

**Loading pretrained weights with timm:**
```python
import timm

model = timm.create_model(
    'vit_base_patch16_224',
    pretrained=True,       # ImageNet-21k weights
    num_classes=22,        # Gravity Spy glitch classes
    drop_rate=0.1,         # Dropout for regularization
)
```

## Training Configuration

| Hyperparameter | Recommended Value | Rationale |
|---------------|-------------------|-----------|
| Optimizer | AdamW | Standard for ViT; decoupled weight decay |
| Learning rate | 5e-5 (transformer) / 1e-3 (head) | Lower lr preserves pretrained features |
| Weight decay | 0.05 | Moderate regularization |
| Batch size | 64-128 | Fits in 32 GB VRAM with AMP |
| Epochs | 80 (early stop at 10 patience) | Fine-tuning converges fast |
| LR schedule | Cosine annealing with 5-epoch warmup | Standard for ViT training |
| Mixed precision | torch.cuda.amp (FP16) | 2x memory savings, faster training |
| Augmentation | RandAugment, RandomResizedCrop, horizontal flip | Spectrograms benefit from geometric augments |
| Label smoothing | 0.1 | Reduces overconfidence, helps noisy labels |
| Class weighting | Inverse frequency or focal loss | Critical for rare class performance |

### Handling Class Imbalance

The Gravity Spy dataset has extreme class imbalance (Blip: ~30K samples; Paired_Doves: ~50 samples). Three strategies, all of which should be combined:

1. **Weighted loss function:** Use `torch.nn.CrossEntropyLoss(weight=class_weights)` where weights are inverse class frequency, capped at 10x the minimum weight to prevent instability.
2. **Oversampling:** Use `WeightedRandomSampler` in the DataLoader to oversample rare classes per epoch.
3. **Focal loss:** Replace CrossEntropy with focal loss (gamma=2.0) to down-weight easy/common examples. Use the `torchvision.ops.sigmoid_focal_loss` or implement directly.

## Anti-Approaches

| Anti-Approach | Why Avoid | What to Do Instead |
|---------------|-----------|-------------------|
| Training ViT from scratch (no pretraining) | Dataset too small (~10^5); ViT needs pretraining to learn attention patterns | Fine-tune from ImageNet-21k pretrained weights |
| Using raw time-series input to ViT | Loses time-frequency structure; ViT expects 2D patch structure | Convert to spectrograms first; ViT operates on images |
| Using 16 kHz strain data | 4x storage/compute cost, frequency content >2 kHz rarely needed for glitch classification | Use 4 kHz downsampled data from GWOSC |
| Training on full O3 strain (not just glitch segments) | Wastes compute on noise-only data; glitches are rare events | Extract glitch segments using Gravity Spy trigger times |
| Using accuracy as primary metric | Misleading under class imbalance -- 95% accuracy possible by ignoring rare classes | Use macro-averaged F1, per-class recall, confusion matrix |
| Resizing Gravity Spy 140x170 images with nearest-neighbor interpolation | Introduces aliasing artifacts | Use bilinear or bicubic interpolation to 224x224 |
