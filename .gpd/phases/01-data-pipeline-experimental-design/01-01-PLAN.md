<objective>
Acquire the Gravity Spy O3 metadata from Zenodo, generate Q-transform spectrograms from GWOSC strain data at native 224x224 resolution, and document per-class sample distributions.

Purpose: Establish the foundational dataset that all downstream model training and evaluation depends on. By generating spectrograms from strain data (rather than using pre-made images that may not cover O3), we ensure full coverage of O3 glitches with controlled normalization at the correct ViT input resolution.
Output: Filtered metadata CSV, generated spectrogram images (224x224, 4 durations per glitch), per-class distribution JSON, download and processing scripts.
</objective>

<task type="auto">
  <name>Task 2: Generate Q-transform spectrograms from GWOSC strain data</name>
  <files>scripts/03_generate_spectrograms.py</files>
  <action>
    Write `scripts/03_generate_spectrograms.py` to generate Q-transform spectrograms from GWOSC O3 strain data for all glitches in the filtered metadata.

    **Step 1: Strain segment discovery and download**

    1. Load `data/metadata/gravity_spy_o3_filtered.csv`
    2. For each unique (ifo, event_time) pair, determine the required strain segment:
       - Need at minimum: [event_time - 8s, event_time + 8s] for the 4.0s duration view (plus whitening padding)
       - Add 4s padding on each side for whitening: total segment = [event_time - 12s, event_time + 12s]
    3. Use `gwosc` to find the strain file URL covering each GPS time:
       ```python
       from gwosc.locate import get_urls
       urls = get_urls('O3a_4KHZ_R1', detector='H1', segment=(gps_start, gps_end))
       ```
       Try O3a_4KHZ_R1 first; if GPS time is in O3b range, use O3b_4KHZ_R1.
    4. Group glitches by their containing strain file to minimize downloads:
       - Each GWOSC HDF5 file covers ~4096s
       - Multiple glitches may share the same strain file
    5. Download strain files to `data/raw/strain/{ifo}/` with resume capability
       - Do NOT download all O3 strain data -- only files containing labeled glitches
       - Expected: ~200-500 unique strain files (not 300-400 GB)
       - Log progress: "Downloading file N/M for {ifo}"
    6. Implement checkpointing: save a `data/raw/strain/download_progress.json` tracking which files are complete. If the script is interrupted and restarted, skip already-downloaded files.

    **Step 2: Q-transform spectrogram generation**

    For each glitch in the filtered metadata:

    1. Load the relevant strain segment using GWpy:
       ```python
       from gwpy.timeseries import TimeSeries
       ts = TimeSeries.read(strain_file_path, start=gps_start, end=gps_end)
       ```
    2. Whiten the strain data (CRITICAL -- unwhitened spectrograms are dominated by detector noise curve):
       ```python
       whitened = ts.whiten(fftlength=4, overlap=2)
       ```
    3. Generate Q-transform for each of the 4 durations, centered on event_time:
       ```python
       for duration in [0.5, 1.0, 2.0, 4.0]:
           half = duration / 2
           qgram = whitened.crop(event_time - half, event_time + half).q_transform(
               qrange=(4, 150),
               frange=(10, 2048),
               logf=True,
               outseg=(event_time - half, event_time + half)
           )
       ```
    4. Normalize: clip to [0, 25.5], then min-max scale to [0, 1]:
       ```python
       import numpy as np
       data = np.clip(qgram.value, 0, 25.5) / 25.5
       ```
    5. Render to 224x224 PNG image:
       - Use matplotlib to render the Q-transform data as an image at exactly 224x224 pixels
       - Use a colormap consistent with Gravity Spy (viridis or similar)
       - No axes, labels, or whitespace -- pure spectrogram image
       - Save as RGB PNG (single-channel replicated to 3 channels per CONVENTIONS.md)
    6. Save to: `data/spectrograms/{ml_label}/{gravityspy_id}_{duration}s.png`

    **Step 3: Parallelization and checkpointing**

    1. Use joblib.Parallel with n_jobs=12 (or os.cpu_count()) for parallel Q-transform generation
    2. Implement per-glitch checkpointing:
       - Save progress to `data/spectrograms/generation_progress.json` every 100 glitches
       - On restart, skip glitches whose 4 spectrogram files already exist
    3. Log throughput: glitches/minute, estimated time remaining
    4. Handle failures gracefully:
       - If a strain segment cannot be loaded (gap in data, corrupted file): log the gravityspy_id and skip
       - If Q-transform fails (e.g., segment too short after whitening): log and skip
       - Track failure count; if > 10% of glitches fail, raise an ERROR

    **Step 4: Post-generation validation**

    1. Count generated spectrograms vs expected (filtered metadata rows x 4 durations)
    2. Report success rate: expect >= 90% of glitches to have all 4 spectrograms
    3. Sample 100 random images and verify dimensions are exactly 224x224x3
    4. Sample 5 high-SNR glitches from common classes and save enlarged comparison plots to `figures/spectrogram_qa/` for visual QA in the SUMMARY

    **Compute budget:**
    - ~8-10K glitches x 4 durations = ~32-40K Q-transforms
    - At ~1-2 seconds per Q-transform = ~9-22 hours single-threaded
    - With 12-core parallelism = ~1-2 hours wall time for generation
    - Strain file downloads: ~50-200 GB depending on file grouping (much less than full O3)
    - Total wall time including downloads: ~2-4 hours
    - AI active time: ~50 min (script writing, launch, monitoring, QA)
  </action>
  <verify>
    1. Spectrogram generation success rate: >= 90% of filtered metadata rows have all 4 duration spectrograms
    2. Image dimensions: 100/100 sampled images are exactly (224, 224, 3)
    3. Pixel values: all in valid range [0, 255] for PNG; no all-black or all-white images in sample
    4. Q-transform parameters match conventions: qrange=(4,150), frange=(10,2048), 4 durations (0.5, 1.0, 2.0, 4.0)
    5. Whitening applied: `whitened = ts.whiten()` confirmed in code path (not raw strain)

    9. QA plots: 5 spectrogram comparison figures saved to figures/spectrogram_qa/
  </verify>
