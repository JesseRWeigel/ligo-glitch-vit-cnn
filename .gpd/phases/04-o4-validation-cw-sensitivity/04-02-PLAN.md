  <action>
    STEP 1: Identify CW-critical classes and their O4 ViT-CNN advantage.
    - Load o4_comparison_table.csv from Plan 04-01.
    - Define CW-critical classes based on spectral characteristics and CW band overlap (20-2000 Hz):
      * HIGH CW impact: Scattered_Light (10-120 Hz), Violin_Mode (~500 Hz + harmonics), Low_Frequency_Lines (10-100 Hz)
      * MEDIUM CW impact: 1080Lines (~1080 Hz), Whistle (200-4000 Hz), Power_Line (60 Hz + harmonics)
      * LOW-MEDIUM: Low_Frequency_Burst (10-50 Hz)
    - For each CW-critical class, record: N_train(O3), F1_vit(O4), F1_cnn(O4), f1_diff_o4.
    - Categorize: "ViT-advantaged CW classes" (f1_diff_o4 > 0) vs "CNN-advantaged CW classes" (f1_diff_o4 < 0).
    - Document the overlap (or disconnect) between ViT advantage and CW relevance. Per RESEARCH.md, Power_Line (N_train=1582, O3 ViT advantage=+0.507) is the strongest candidate. Most other CW-critical classes favor CNN on O3.

    STEP 2: Generate per-class veto segments for both models.
    - Load O4 predictions from results/04-o4-validation/o4_predictions.npz (y_true, cnn_preds, vit_preds, cnn_confidences, vit_confidences).
    - Load O4 manifest from data/o4/metadata/o4_evaluation_manifest.csv (GPS times, durations).
    - For each CW-critical class and each model:
      a. Select glitches where the model predicts this class with confidence >= threshold (default: 0.7).
      b. Create veto segments: [gps_time - duration/2, gps_time + duration/2] for each selected glitch.
      c. Merge overlapping segments.
    - Create TWO veto segment lists: vit_veto_segments and cnn_veto_segments, each covering all CW-critical classes.
    - Also create per-class veto segment lists for individual class analysis.

    STEP 3: Obtain O4a science mode segments.
    - Use gwpy.segments.DataQualityFlag or gwosc package to get O4a science mode segments for the GPS range covered by the O4 evaluation data.
    - If GWOSC DQ flags are not available programmatically, approximate T_total as the GPS time span of the evaluation set.
    - T_total = total duration of science-mode observation time in the evaluation window.

    STEP 4: Compute duty cycle for each veto strategy.
    - For each model (ViT, CNN) and each class grouping (all CW-critical, ViT-advantaged CW only, per-individual-class):
      a. T_vetoed = total duration of veto segments intersecting with science-mode time.
      b. T_clean = T_total - T_vetoed.
      c. DC = T_clean / T_total.
    - Compute delta_DC = DC_vit - DC_cnn (positive means ViT vetoes remove less observation time = better for CW searches, OR ViT vetoes remove the RIGHT glitches more efficiently).
    - Compute bootstrap CI on delta_DC: resample glitch-level predictions 10K times, regenerate veto segments, recompute DC each time.

    STEP 5: Compute veto efficiency.
    - For each model: among glitches with ground-truth label in CW-critical classes, what fraction does the model correctly flag?
    - eta_veto = N_correctly_flagged / N_total_cw_glitches.
    - tau_dead = T_vetoed / T_total (deadtime fraction).
    - The KEY metric is the tradeoff: how much glitch contamination removed per unit observation time lost.

    STEP 6: Save results.
    - Save per-class duty cycle comparison to results/04-o4-validation/cw_duty_cycle_comparison.csv.
    - Save comprehensive results to results/04-o4-validation/cw_veto_results.json including:
      * duty_cycle_vit, duty_cycle_cnn, delta_dc, delta_dc_ci
      * per_class_cw_breakdown (for each CW-critical class: vit_dc, cnn_dc, delta_dc, n_glitches, vit_advantage_on_o4)
      * veto_efficiency_vit, veto_efficiency_cnn
      * deadtime_vit, deadtime_cnn
      * cw_critical_classes (list with CW relevance level)
      * vit_advantaged_cw_classes, cnn_advantaged_cw_classes

    IMPORTANT: If delta_DC is < 0.001 (one part per thousand) for all CW-critical classes combined, this is the backtracking trigger for claim-cw-benefit. In this case:
    - Try narrowing to individual CW-critical classes (especially Power_Line).
    - Try frequency sub-band analysis (e.g., duty cycle improvement in 50-70 Hz band for Power_Line).
    - If still < 0.001, report as null result with honest interpretation.
  </action>

    3. Veto efficiency in [0, 1] for both models.
